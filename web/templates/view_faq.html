About this PoC
A 3-day de-risking spike for the AnyBank multi-cloud cost engagement. The point is to show what a FOCUS-conformant console can and can't answer — honestly — before AnyBank's team builds the real thing. Every figure here is synthetic (DEMO- prefixed) and USD-normalised. This page answers the questions we get asked most.

01.The basics
Is this real AnyBank data?

No. Every number, account ID, resource name and workload is synthetic and prefixed DEMO- so it can never be mistaken for production. Account IDs are fake, costs are generated, and no real cloud credentials or AnyBank systems are connected. The shapes are realistic; the values are invented.

What does this PoC actually demonstrate?

That multi-cloud cost (AWS, Azure, OCI) plus on-prem can be normalised to the FOCUS v1.3 open spec, joined to ManageIQ inventory and utilization, and presented against four leadership requirements — each with an honest verdict on whether FOCUS handles it natively, partially, conditionally, or not at all. Verdicts before numbers.

Why USD when AnyBank reports in AED?

AED is pegged to USD at 3.6725, so we normalise every provider's cost to USD as a single reporting currency. Mixing currencies in a total is the classic silent error — Azure bills in AED, AWS/OCI in USD. The console always shows the USD-normalised figure and labels the source currency separately so nothing is mixed.

02.The data & the join
Why are only some workloads joined to inventory?

The cost-to-inventory join is the hard part. FOCUS rows carry a ResourceId; ManageIQ carries its own keys — and they're asymmetric across providers (AWS/OCI match on uid_ems, Azure on ems_ref). Some rows are cost-only (a managed service with no VM), some are inventory-only (no cost row yet). We surface the residual rows rather than silently dropping them — the join distribution on the Overview shows exactly what matched and what didn't.

Where does the on-prem cost come from?

FOCUS doesn't source on-prem data. The on-prem half is a per-resource recharge model (a vCPU rate + a memory-GB rate per month) applied to ManageIQ inventory. The rates here are illustrative stubs — AnyBank's chargeback owner substitutes the real numbers. It's labelled "estimated" wherever it appears.

Can I trust the per-workload costs?

Within the synthetic dataset, yes — they reconcile. Each workload's cost comes from the pre-summed, USD-normalised join record, and the same figure appears on the Overview, the Utilization view, the drill-down, and the AI answers. If two screens ever disagree, that's a bug we'd want to know about — they're built to agree.

Is AI / GenAI spend supported by FOCUS? How is it broken down by model?

Partly — and the console says so honestly ("Conditional"). AI cost is FOCUS-native: ServiceCategory = 'AI and Machine Learning' is a mandatory allowed value, and FOCUS v1.3 adds ServiceSubcategory values including Generative AI and Machine Learning. So "how much are we spending on AI?" is fully spec-supported.

What is NOT yet normative is the breakdown by model and token type. FOCUS has no column for "claude-3-5-sonnet" or "input vs output tokens." We carry the model id in SkuMeter (the general "what's being metered" field) and parse it out — Bedrock and Azure OpenAI populate it cleanly; OCI Gen-AI is inconsistent and needs backfill. That provider-dependence is exactly why the verdict is "Conditional," not "Native."

This isn't a gap we invented — the FOCUS community is actively standardising it right now (open proposals for recommended AI SkuMeter values, an InputModality property, and token/cache representation). The proposals extend SkuMeter, so this approach ages well: when ratified, it becomes a mapping change, not a redesign.

Could we get richer AI data from an AI gateway instead of the billing export?

Yes — and for per-model / per-team / per-prompt detail, that's the stronger source. The FOCUS billing export answers "what did AI cost," but an AI gateway (e.g. a Bedrock/LiteLLM-style proxy that all AI calls route through) sees what billing can't: the exact model, input/output token counts, the calling application or team, latency, and cache hits — per request, in real time.

The intended pattern is both, joined — not either/or: the FOCUS export stays the authoritative cost (it's what you're actually invoiced), and the gateway provides the usage dimensions (model, tokens, team) to attribute that cost — the same shape as the FOCUS↔ManageIQ join elsewhere in this console (cost from one source, the breakdown dimension from another). It also future-proofs: if AnyBank standardises AI access behind a gateway, the per-model and chargeback story stops depending on how each provider tags SkuMeter.

It's a roadmap option, not built in this PoC — but it's the natural next step for AI cost attribution and worth scoping in the sprint.

Why is "cost by business unit" mostly untagged?

Business unit isn't a FOCUS column — it lives in resource tags (cost-center), and tagging is incomplete in this synthetic set (~38%). The console shows the tagged breakdown and the untagged remainder rather than hiding the gap. In production, tag-coverage discipline is exactly the kind of thing this surfaces.

03.The AI assistant
How does "Ask the data" work — and can it give a wrong number?

You ask in plain language; the model writes a read-only SQL query, which is parsed by an allowlist guard (single SELECT against four tables, nothing else) before it runs. The answer is then built deterministically from the rows the query returned — not narrated freehand — so it can't state a figure the data doesn't contain. Every answer shows the exact SQL it ran. The optional "Interpret & recommend" layer adds prose, but it too is bounded to those returned rows and will refuse to invent a number that isn't there.

What model is it, and where does my question go?

Amazon Bedrock, Claude Sonnet, via the us-east-1 cross-region inference profile. For this synthetic demo the question text leaves the Gulf to a US commercial region — acceptable because the data is fake. For real AnyBank data this is a residency decision for legal (me-central-1 has no on-demand Claude today; the alternative is the global inference profile). The console works fully with the AI turned off — the canned queries never touch Bedrock.

What happens if the AI is unavailable?

It fails closed and says so plainly. The canned queries still run (they're plain SQL), and every deterministic answer stands on its own. We'd rather show "AI off · canned only" than a confident wrong answer — for a bank, a wrong-cost-with-confident-narration is the worst outcome, so the whole layer is built to avoid it.

04.Carbon, scope & what's next
Why is carbon a "roadmap" and not a number?

FOCUS has no carbon, emissions, or energy column through v1.4. A carbon figure derived from cost alone would be confidently wrong. Real attribution needs per-provider feeds (AWS CCFT, Azure Emissions Impact Dashboard) that are account- or subscription-level and monthly — plus a custom on-prem model. The Carbon view shows the four data streams and their honest status, not a fabricated tonnage.

Is this production-ready?

No — it's a de-risking spike. It runs as a one-command Docker stack and is portable to AWS or on-prem. The production architecture (ROSA/OpenShift + Aurora, connector framework, Secrets Manager) is designed and documented, not deployed. The real deliverable is the captured set of gotchas so AnyBank's team builds the real thing faster.

How would AnyBank connect a real cloud account?

Via the Connect page — onboarding a source is a registry entry plus a credential reference, never a code change. In this PoC a registered source reads a synthetic export; in production the same adapter reads the provider's real FOCUS export from S3/blob using a Secrets Manager credential. The dispatcher normalises it and the join runs automatically.

How do I report something that looks wrong?

Note the page and the figure and raise it with the AWS engagement team. Because the data is synthetic, "looks wrong" usually means either a deliberate honest-gap we're highlighting (an unmatched row, an untagged cost) or a genuine bug — both are useful feedback for the sprint.
